IoT Fire Detection System

Problem & Context: Detecting early-stage fire in agricultural fields is challenging because small smoke bursts and environmental noise must be ignored while real risk is captured. The goal of the system was to combine raw sensor data with tuned thresholds to generate meaningful alerts only when needed.

Architecture & Stack: AQI, smoke, and temperature sensors were interfaced with a microcontroller, and multi-sensor fusion logic was implemented where each reading was mapped to calibrated ranges. Processing was done with C-style logic: per-sensor weighting, a combined risk score, and tiered alert levels (safe, caution, critical) that could trigger hardware outputs or notifications.

Key Learnings: The project showed that production-like systems depend not only on a single threshold but on noise filtering, debounce logic, and environment-specific tuning. This experience forms a base for future IoT or embedded work where reliability and control over false positives directly influence user trust.

Algorithm Visualizer

Objective: Move DSA concepts beyond notebook pseudocode by turning C implementations into real-time visuals, so comparisons, swaps, path choices, and backtracking steps are clearly visible on screen.

Implementation: Using C with Raylib, a modular engine was built that maps sorting algorithms (Bubble, Selection, Insertion, Merge, Quick, Radix, Bucket, Cocktail), Dijkstra‚Äôs shortest path, and N-Queens backtracking into visual states. Each algorithm has its own update‚Äìrender loop, making time complexity visible through bar heights, colors, and animation speed.

Impact & Learning: The visualizer changed the debugging style: instead of only catching segmentation faults or wrong outputs, it became possible to catch wrong frame transitions, stuck states, and unnatural animations. This improved both reasoning and code quality and serves as a strong portfolio piece that demonstrates DSA understanding at an execution level, not just theoretical.

Tourist Reservation System (C++)

System Design: A console-based C++ application with clearly separated admin and user flows, implemented using OOP pillars (abstraction, inheritance, polymorphism, encapsulation). Core entities such as User, Booking, and Package were modeled as classes with well-defined responsibilities.

Features & Flow: The system supports booking creation, search, cancellation, and simple payment simulations, with all operations persisted via file-based storage. Input validation, duplicate booking checks, and edge-case handling keep user journeys predictable and stable, even without a graphical interface.

Outcome: This project introduced production-style thinking: dealing with logs, error messages, and data consistency while breaking logic into reusable functions and classes. It acts as a strong starting point for backend or systems-oriented roles.

Postman Library API v2

API Scope: A library-style REST API where CRUD operations for books and users were designed with clear endpoints, request bodies, and status codes. The focus was on predictable contracts and clean responses, not just basic connectivity.

Testing & Workflow: Postman collections, environments, and test scripts were used to cover both happy-path and negative scenarios, such as invalid payloads, missing fields, or unauthorized requests. After each change, the collection was re-run to catch regressions early.

Professional Learning: The project helped develop an ‚ÄúAPI-first‚Äù mindset‚Äîprioritising documentation, consistent naming, and disciplined status-code usage. These habits transfer directly to real-world backend or API testing work, where reliability and clarity shape team collaboration.
